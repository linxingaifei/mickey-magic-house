# GitHub 用户 10 分钟上手指南（米奇妙妙屋）

这份文档给「先用 GitHub、后接触本软件」的同学使用。
目标：不讲复杂术语，10 分钟完成最常见操作。

---

## 1. 先建立一个“映射关系”

- GitHub 的 **Issue/PR 描述** ≈ 这里的「任务输入」
- GitHub 的 **Files changed / Diff** ≈ 这里的「修改结果」
- GitHub 的 **Actions 日志** ≈ 这里的「命令输出/执行日志」
- GitHub 的 **Commit 历史** ≈ 这里的「会话历史 + 提交记录」

把它理解成：
> 提需求、执行、看结果，被集中到同一界面里。

---

## 2. 第一次使用：最小流程（建议照着做）

1. 打开项目并确认目录（例如 `mickey-magic-house`）
2. 在输入区写一句目标（例如：`帮我新增一份部署文档`）
3. 等待执行完成后，先看「修改了哪些文件」
4. 再看「输出日志」是否报错
5. 最后做一次提交（commit）

> 一开始不要追求全会，用“一个目标一轮”最容易上手。

---

## 3. 你最常用的三个区域

### A. 会话/任务区（左侧或顶部）
用途：回看历史、切换上下文。

### B. 输入区（底部）
用途：提需求、补充限制条件。

推荐写法：
- 目标：要做什么
- 范围：改哪些文件/模块
- 验收：你希望看到什么结果

示例：
```text
目标：补一份 OpenWrt 编译文档
范围：docs/ 目录，README 增加链接
验收：文档含环境准备、编译、常见报错
```

### C. 输出与变更区（中间）
用途：看命令结果、看改动、看报错。

遇到报错时，优先看：
1. 最后一条失败命令
2. 失败前后 20 行日志
3. 哪个文件报错

---

## 4. 浏览器与客户端如何同步

如果你在浏览器做过操作，想在客户端继续：

1. 浏览器与客户端登录同一账号
2. 确认开启历史记录（不要用临时/无痕会话）
3. 客户端刷新或重新登录一次
4. 仍不同步时，检查客户端版本并升级

> 关键点：同步依赖账号云端历史，不依赖本地浏览器缓存。

---

## 5. 给 GitHub 用户的“低成本习惯”

- 每次只提一个明确目标
- 每轮结束都看一次变更列表
- 重要结果及时 commit
- 关键文本（配置、命令）额外复制到你常用笔记

---

## 6. 常见不习惯 & 对应解法

### Q1：界面不熟，容易慌
解法：只盯三个区域（会话、输入、输出），先稳定流程。

### Q2：不知道该怎么提需求
解法：按「目标/范围/验收」三行模板写。

### Q3：担心改坏
解法：先让系统展示变更，再决定是否提交。

### Q4：看不到之前会话
解法：先检查账号一致，再检查是否开启历史记录。
